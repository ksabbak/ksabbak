We have reached the middle of the SOLID principles, the L! Which stands for Liskov Substitution Principle. Well that's super clear, everyone go home. I lied, let's look at what this means:

Subtype Requirement: Let ɸ(x) be a property provable about objects x of type T. Then ɸ(y) should be true for objects y of type S where S is a subtype of T.

I mean it's not that bad, I guess? The notation is a bit scary, but the idea is that as long as all instances of a class has something that's always going to be the same, then all instances of that class's subclass is going to have to have that same something that's also always the same. But only if it's "provable" aka public (or protected, but not private). Hm. I think Liskov said it better. Let's try again with some examples.

So what we're actually going for here, is that if we use the parent class in our code and then write a child class, any time we use an instance of the parent class, we have to be able to substitute it with an instance of the child class without anything breaking. You don't have to get the same exact results, but your code can't collapse. I'm going to illustrate this with examples from a terrible movie: the 2003 remake of Freaky Friday.

So we have a parent class, Jamie Lee Curtis.

{% highlight ruby %} class JamieLeeCurtis; end {% endhighlight %}

We have a child class, Lindsay Lohan.

{% highlight ruby %} class LindsayLohan < JamieLeeCurtis; end {% endhighlight %}

So far so good. Now, early on in the movie, they switch bodies. Which means that every place that the parent class (Jamie Lee Curtis) is used, we get Lindsay Lohan instead. Now, they both have basically the same functionality. Both Jamie Lee Curtis and Linsay Lohan can walk and talk and drive and be a therapist and talk about a book and while Lindsay Lohan is not actually good at most of those things, she's able to perform the tasks.

{% highlight ruby %} class JamieLeeCurtis def walk() "normally" end def talk() "Slightly pretentiously" end def psychoanalyze() "well" end def drive() "normally" end end {% endhighlight %}
{% highlight ruby %} class LindsayLohan < JamieLeeCurtis def walk() "normally" end def talk() "Like a teen girl" end def psychoanalyze() "poorly" end def drive() "barely passably" end end {% endhighlight %}

As we can see, so far so good. Maybe we don't want someone who drives like a disaster, but that's why our original code had an instance of the parent class. We can still substitute the child class and it will get from point A to point B. Or, in the code example, it still takes the same argument types and still returns something of the same type.
